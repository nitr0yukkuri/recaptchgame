# reCAPTCHA風ゲームで、相手の画面に玉ねぎを降らせるまでのmotion設計

## はじめに

こんにちは！わかと（@0st_ts）です。

今回は、reCAPTCHA風のリアルタイム対戦ゲームで、相手の画面に玉ねぎを降らせるまでのmotion設計について書きます。

作ったものは、reCAPTCHAのように「車」「信号機」「階段」などのお題に合う画像を選択していく対戦ゲームです。

ただ画像を選ぶだけだと、普通の画像選択ゲームになってしまいます。

なので今回は、正解を続けるとコンボがつながり、相手の画面に妨害エフェクトを送れるようにしました。

妨害エフェクトには、

* SHAKE
* SPIN
* BLUR
* INVERT
* GRAYSCALE
* ONION_RAIN

などがあります。

その中でも一番意味が分からないのが、相手の画面に玉ねぎを降らせる `ONION_RAIN` です。

<img src="https://raw.githubusercontent.com/nitr0yukkuri/recaptchgame/main/docs/images/onion-rain-obstruction.png" alt="ONION_RAINで相手の画面に玉ねぎが降っている様子" width="700">

この記事では、ただネタとして玉ねぎを降らせた話ではなく、対戦ゲームにおけるmotionをどう考えたかを書きます。

## 作ったもの

作ったものは、reCAPTCHA風のリアルタイム対戦ゲームです。

お題が表示されて、9枚の画像の中から該当するものを選択します。

例えば、

> 以下の画像をすべて選択：車

と表示されたら、車の画像をすべて選択して確認します。

正解するとスコアが入り、相手より先に目標スコアに到達した方が勝ちです。

また、相手がどの画像を選択しているかを見られる `RIVAL VIEW` も用意しています。

これによって、ただ自分の画面だけを見るのではなく、相手の進行状況も見ながら焦るような体験になります。

## 使った技術

主に使った技術は以下です。

```txt id="mqi63r"
React
TypeScript
Vite
Zustand
Tailwind CSS
Framer Motion
WebSocket
Go
```

フロントエンドではReactとTypeScriptを使っています。

状態管理にはZustandを使い、ゲームの状態、スコア、コンボ数、相手の選択状況、妨害エフェクトなどを管理しています。

motion周りはFramer Motionを使っています。

バックエンドはGoで、WebSocketを使ってリアルタイムに対戦状態を同期しています。

## なぜmotionを入れたのか

最初に考えたのは、reCAPTCHA風のゲームは見た目だけだと地味になりやすいということです。

画像を選ぶ。
確認する。
正解する。
スコアが増える。

これだけでもゲームとしては成立します。

ただ、対戦ゲームとして考えると、少し盛り上がりが足りないと感じました。

相手より早く選ぶ緊張感はありますが、画面上の変化が少ないと、コンボが続いても「強くなっている感」が出にくいです。

そこで、コンボがつながったときに相手の画面へ妨害エフェクトを送るようにしました。

ここでのmotionは、ただ気持ちよくするためのものではありません。

むしろ、相手を少し焦らせるためのmotionです。

## motionの目的

今回のmotionで意識したのは、以下の5つです。

* 相手を焦らせる
* でも理不尽すぎない
* 対戦の盛り上がりを作る
* 画面の変化でコンボの強さを伝える
* ネタとして記憶に残す

普通のUIアニメーションだと、操作したときに気持ちよく見えることや、画面遷移を自然に見せることが大事になります。

でも今回のゲームでは、motionの役割が少し違います。

このゲームにおけるmotionは、相手の集中を少しだけ崩すためのものです。

ただし、完全に操作不能にしてしまうと、ゲームとして理不尽になります。

なので、見た目はかなり邪魔だけど、頑張ればプレイは続けられるくらいのラインを狙いました。

## 妨害エフェクトの種類

妨害エフェクトは、画面の見え方や動き方を変えることで、相手にプレッシャーをかけるものです。

ざっくり分類すると、こんな感じです。

```txt id="mazztd"
SHAKE      -> 画面を揺らして焦らせる
SPIN       -> 画面を回転させて混乱させる
BLUR       -> ぼかして見づらくする
INVERT     -> 色を反転させて認識しづらくする
GRAYSCALE  -> 色情報を消して判別しづらくする
ONION_RAIN -> 玉ねぎを降らせて意味不明にする
```

それぞれ妨害の方向性が違います。

`SHAKE` は、操作自体はできるけど視点が安定しません。

`SPIN` はかなり強めで、画面の向きが変わるので混乱します。

`BLUR` は画像認識そのものを邪魔します。

`INVERT` や `GRAYSCALE` は、画像の色から判断しづらくします。

そして `ONION_RAIN` は、ゲーム的な合理性よりも、見た瞬間に笑えることを優先しています。

## Framer Motionで動きを分ける

動きのあるエフェクトは、Framer Motionのvariantsで管理しています。

例えば、`SHAKE` や `SPIN`、`SKEW` のような動きは、motionのvariantsとして定義できます。

```ts id="vel9ix"
const obstructionVariants = {
  SHAKE: {
    x: [-15, 15, -15, 15, 0],
    transition: {
      repeat: Infinity,
      duration: 0.5,
    },
  },
  SPIN: {
    rotate: 360,
    transition: {
      repeat: Infinity,
      duration: 0.5,
      ease: "linear",
    },
  },
  NORMAL: {
    x: 0,
    rotate: 0,
    skewX: 0,
  },
}
```

一方で、`BLUR` や `INVERT`、`GRAYSCALE` は、motionで座標を動かすというより、CSSのfilterを使った方が分かりやすいです。

なので、すべてをFramer Motionで無理に表現するのではなく、

* 動きで邪魔するもの
* 見た目のfilterで邪魔するもの
* オーバーレイで邪魔するもの

に分けました。

この分け方は結構大事でした。

全部をmotionでやろうとすると、コードも見た目も複雑になります。

逆に、filterで済むものはCSSに任せた方がシンプルです。

## ONION_RAINをどう作ったか

`ONION_RAIN` は、画面の上から玉ねぎ画像が大量に降ってくるエフェクトです。

名前の通り、玉ねぎの雨です。

実装としては、玉ねぎ画像を複数生成し、それぞれにランダムな位置、遅延、サイズ、落下速度を持たせています。

```ts id="0se5e9"
const onions = Array.from({ length: 100 }).map((_, i) => ({
  id: i,
  left: Math.random() * 100,
  delay: Math.random() * 2,
  duration: 0.5 + Math.random() * 1.5,
  size: 30 + Math.random() * 100,
}))
```

これによって、全部の玉ねぎが同じ位置・同じ速度で落ちるのではなく、バラバラに降ってくるようになります。

もし全ての玉ねぎが同じ動きをしていたら、ただの背景アニメーションっぽくなってしまいます。

でもランダム性を入れると、一気に「なんか画面が荒らされている感」が出ます。

玉ねぎ自体は、上から下に落ちて、ついでに回転します。

```ts id="z5n9ej"
initial={{ y: -150, opacity: 1, rotate: 0 }}
animate={{ y: 800, rotate: 720 }}
transition={{
  duration: o.duration,
  repeat: Infinity,
  delay: o.delay,
  ease: "linear",
}}
```

やっていることはシンプルです。

でも、対戦中にこれを相手に送ると、かなり意味が分からない画面になります。

この「意味が分からない」が大事でした。

## なぜ玉ねぎなのか

正直、玉ねぎである必要はありません。

ただ、ゲーム内の画像素材として玉ねぎがあり、画面に降らせたときの見た目がかなり強かったです。

かっこいいエフェクトにしようと思えば、雷やノイズや炎でもよかったと思います。

でもそれだと、普通のゲームの妨害エフェクトになります。

今回作りたかったのは、ただ強い妨害ではなく、見た人の記憶に残る妨害でした。

`SHAKE` や `BLUR` は、ゲームとして分かりやすい妨害です。

一方で、`ONION_RAIN` はゲーム的な意味が分からないです。

でも、その意味の分からなさが、reCAPTCHA風ゲームの少し変な世界観に合っていると感じました。

## 理不尽すぎないようにする

妨害エフェクトを入れるときに一番気をつけたのは、理不尽になりすぎないことです。

対戦ゲームで相手の画面を邪魔できるのは楽しいです。

でも、邪魔が強すぎると、された側はただ不快になります。

特にreCAPTCHA風ゲームは、画像を見て判断するゲームです。

そのため、見え方を邪魔しすぎると、ゲームの根本を壊してしまいます。

例えば、`BLUR` を強くしすぎると何も見えません。

`SPIN` を速くしすぎると操作どころではありません。

`ONION_RAIN` も、玉ねぎの数やサイズを増やしすぎると画面が完全に埋まります。

なので、妨害はあくまで「焦らせる」ためのものとして考えました。

完全にプレイ不能にするのではなく、

> うわ、邪魔。でもまだいける。

くらいがちょうどいいと思っています。

## 自分側と相手側で扱いを変える

実装していて面白かったのは、自分の画面と相手の画面で、エフェクトの扱いを少し変える必要があったことです。

相手側の画面には、`RIVAL VIEW` として相手の選択状況が表示されます。

そこに妨害エフェクトが反映されると、相手が今どんな状態なのかが視覚的に分かります。

一方で、自分側の画面に強すぎるmotionをそのまま入れると、自分の操作感も壊れてしまいます。

実際、プレイヤー側の画像が意図せず回転しないように、`SPIN` をプレイヤー本体には適用しないようにしています。

このあたりは、ゲームとしての見た目と操作性のバランスでした。

相手を邪魔する演出は入れたい。

でも、自分が操作する部分まで壊すと、ただ遊びにくいだけになります。

## motionは気持ちよさだけじゃない

今回やってみて感じたのは、motionは必ずしも気持ちよくするためだけのものではないということです。

UIアニメーションでは、ユーザーにとって自然で、分かりやすく、気持ちいい動きが求められることが多いと思います。

でもゲームの場合、少し話が変わります。

特に対戦ゲームでは、motionによって相手を焦らせたり、コンボの圧を伝えたり、場の盛り上がりを作ったりできます。

今回のmotionは、きれいに見せるためというより、対戦の空気を変えるためのものでした。

相手の画面が揺れる。

色が反転する。

ぼやける。

そして玉ねぎが降る。

どれも普通のUIならやらない方がいい動きかもしれません。

でも、ゲームの中ではそれが面白さになります。

## 反省点

反省点としては、エフェクトごとの強さの調整がかなり難しかったです。

`SHAKE` や `SPIN` は分かりやすいですが、少し強くするだけで一気に不快になります。

`BLUR` や `GRAYSCALE` は、見た目の変化は分かりやすいですが、画像選択ゲームとしてはかなり影響が大きいです。

`ONION_RAIN` はネタとして強いですが、玉ねぎの数やサイズ次第で画面を埋めすぎてしまいます。

また、妨害エフェクトは「面白い」と「うざい」の境界がかなり近いです。

作っている側は楽しいですが、遊ぶ側がどう感じるかは別です。

なので、最終的には実際に触って、邪魔だけど笑えるくらいのラインを探る必要があると感じました。

## おわりに

今回は、reCAPTCHA風ゲームで、相手の画面に玉ねぎを降らせるまでのmotion設計について書きました。

motionというと、気持ちいいUIや自然な画面遷移を作るものというイメージがあります。

でも、ゲームではmotionを使って、相手を焦らせたり、対戦の盛り上がりを作ったり、ネタとして記憶に残る体験を作ることもできます。

今回の `ONION_RAIN` は、正直かなり意味が分からないエフェクトです。

でも、相手の画面に玉ねぎが降ってくるというだけで、かなり記憶に残ります。

技術的には、Framer Motionで動きを作り、Zustandで妨害状態を管理し、WebSocketで対戦状態と組み合わせています。

ただ、最終的に大事だったのは、どの技術を使ったかよりも、

> どんな気持ちにさせたいのか

だった気がします。

今回は、相手を少し焦らせる、でも理不尽すぎない、そして見た人の記憶に残るmotionを目指しました。

その結果、相手の画面に玉ねぎを降らせることになりました。
